## 1.0 WORKING WITH CALENDARS

The *Access Calendar* function allows you to create weekly schedules that can then be associated with
one or multiple permissions. Calendars define periods of time that dictate rules for both access control
and workstation access. All personnel assigned to a permission level are subject to the access hours
defined in the associated calendars.
For example, if a permission level allows access to a specific building at all hours of the day and on any
day of the week, you will need to create a calendar reflecting this 24 x 7 coverage. If a second
permission level will only allow access to the building during the normal Monday through Friday, 8:00
to 6:00 workday, you will need a second calendar reflecting this timeframe. Permissions that follow
other access rules will require additional calendars. The number of calendars needed depends on the
complexity of your overall access control plan.

## 2.0 SETTING ACCESS PERMISSIONS

All *Access Permissions* in the system are accomplished via the creation of *Access Rules*. A rule can be
thought of as a sentence that answers the questions of *Who*, *Where*, and *When*.
Example 1 - John Smith is allowed through the front door all of the time. In this case:

The *Who* is ‐
John Smith

The *Where* is ‐
front door

The *When* is  ‐
all of the time
Example 2 - Shift 2 employees are allowed through the factory doors between 12:00pm and
8:00pm on working days. In this case:

The *Who* is ‐
shift 2 employees

The *Where* is ‐
factory doors

The *When* is ‐
between 12:00pm and 8:00pm on working days

![Administration Panel image 57](./images/starwatch-sms-user-guide-administration-panel-rev-10_57.png)

## 2.1 WORKING WITH ACCESS GROUPS

*When* aspects of *Access* *Rules* are defined using *Calendars* (see section 1.0 WORKING WITH
CALENDARS).
*Where* aspects of *Access* *Rules* are defined using *Access Groups*. These groups are collections of doors in
the system that all have the same rules applied to them. There are generally no restrictions on how
doors are grouped together in an *Access Group*.
Example 1 - doors that lead to the same *Physical* place, such as factory doors
Example 2 - doors that lead to the same *Type* of place, such as entrances or common areas
Example 3 - doors that share the same set of *People* who have access to them, such as IT doors
The management of *Access Groups* and the doors that are contained in each is accomplished using the
*Access* application.

## 2.2 DEFINING ACCESS PERMISSIONS

*When* aspects of *Access* *Rules* are defined using *Calendars* (see section 1.0 WORKING WITH
CALENDARS). *Where* aspects of *Access* *Rules* are defined using *Access Groups* (see section 2.1 WORKING
WITH ACCESS GROUPS).
Once these aspects have been configured, you can begin adding the *Who* aspects by creating *Access*
*Permissions*. There are three ways to define permissions in the system:

*Person Group* Permissions

*Individual* Permissions

*Card* Permissions
Note: DAQ recommends that *System Administrators* gain a general understanding of each option before
choosing which *Access Permission* path to start with. Depending on the system and its complexity, one
type of permission may be preferable over another. Options may also be used in combination, as
appropriate. For dynamic systems that change rules and add people regularly, DAQ recommends the
*Person Group* permission method. Basic systems can also use this method since it is an easier and
cleaner way to manage your system over time.
Prior to configuring *Access Permissions*, the user should make sure they have done the following:

Create a *Role* to use when logging into the *System Operator* application and assign rights to the
login for viewing, modifying, and accessing various permissions.

Create *Calendars* to achieve a set of time-based access permissions. For example, a *Calendar* for
a visitor might only grant access from 9:00am to 4:00pm, while a *Calendar* for employees might
grant access from 7:00am to 10:00pm.

![Administration Panel image 85](./images/starwatch-sms-user-guide-administration-panel-rev-10_85.png)

## 2.2.1 PERSON GROUP PERMISSIONS

*Person Groups* are groups of people that share a common collection of *Access Rules*. For example, all
people in an employee *Person Group* may be given access to the main entrances and common areas of
a facility, while people in an IT-related *Person Group* might be given access to the IT areas of the
building. Individuals may be placed into one or multiple groups, allowing them to gain access to the
areas they require.
There are several default *Person Groups* provided in the *StarWatch SMS* application. If you do not wish
to use these groups, you can rename or delete them; otherwise, they can be used as a good starting
point for setting up your security system. It should also be noted that while *Person Groups* can and are
used for access control, they can also be used for the purpose of grouping and sorting people in the
system. When a group is created, it is always used for simple grouping by default, and can then be
selected to be used for access control as well.
Note: you cannot delete the *Admin*, *Retired*, or *Visitor* groups, as they are the required minimum
group types that your system must have.

![Administration Panel image 86](./images/starwatch-sms-user-guide-administration-panel-rev-10_86.png)

## 2.2.2 CARD PERMISSIONS

*Card Permissions* provide a method for assigning *Access Rules* to a *Card* instead of to a *Person*. This can
be useful in certain circumstances, but in general is not the simplest way to create *Access Rules* and
permissions, as they can be difficult to manage over time, especially in complex systems. However,
*Card Permissions* can be beneficial when:

The facility has individuals that carry multiple credentials, and are forced to use a specific card
at a specific door. This would mean that each *Card* that the person is carrying has different
rules associated with it, and the user needs to know which card to use at each door.

The facility temporarily issues *Cards* to access certain areas, and then the cards are returned.
Think of these cards as electronic “keys” that are already programmed with a *Card Permission*
to get into the specific area. The cards are given out, people use them for the day, and then
return them. The following day, they are issued out again to different people who require
access.
The complexity of *Card Permissions* is that they require that all of the *Access Rules* be combined
together into a permission. So, if we had an access permission called *Shift 1 Access*, we would need to
define every rule for *Shift 1 Access*.

Shift 1 Access is allowed through the Office Doors 24x7

Shift 1 Access is allowed through the Exterior Building Doors during Shift 1 hours

Etc.
This works nicely in the initial set up of a system, but begins to break down as individuals (or *Cards*)
require different combinations of doors. This is because a *Card* can only have one *Card Permission*
assigned to it. It cannot use combinations.

![Administration Panel image 159](./images/starwatch-sms-user-guide-administration-panel-rev-10_159.png)

*Card Permissions* are used to associate specific *Card Types* with access points. *Card Permissions*
encapsulate everything about where a card can go and when. Each *Card* can only have one *Card*
*Permission*. As mentioned earlier, this option is useful for simple, non-dynamic systems that do not
require many rules for doors.
To create a *Card Permission*, you will need to define a set of rules that can be applied to a specific
*Card* that will be given to a person. Each *Card Permission* defines all the rules associated with every
*Access Group* in the system, including defining when the *Card* can go through the groups of doors it is
allowed to go through and defining the doors it is not allowed to go through.
The management of *Card Permissions* and the rules contained in each is accomplished using the *Access*
*Permissions* tab of the *Access* application.

Maintaining Card Permissions
While initial set up of a system using *Card Permissions* is very straight-forward, it can become
cumbersome as a system becomes more complex and rules change.
As an example, imagine that the facility constructs a training area where we need half of the *Shift 1*
employees and half of the *Shift 2* employees to be able to access the new area during training hours in
addition to the permissions they already have from the assigned *Shift 1* or *Shift 2* access permissions.
Because a *Card Permission* needs to contain all of the *Access Rules* for every door, and a *Card* may only
have one permission attached to it, the following steps are required:

Create a new *Access Permission* for *Shift 1* *+ Training Access*

Create  a new *Access Permission* for *Shift 2 + Training Access*

Move the *Cards* that require the Training Area from *Shift 1* access into *Shift 1 + Training Access*

Move the *Cards* that require the Training Area from *Shift 2* access into *Shift 2 + Training Access*
This means that we need to create another two sets of *Rules* and change their *Card Permissions* to
allow the selected employee groups to have access to the training area in addition to their normal
working hour shifts. Below are the *Card Permission* rules that need to be created:

Rule 1 - Card Permission 1: Shift 1-A Employee: Calendar (8:00am - 4:00pm)

Rule 2 - Card Permission 2: Shift 2-B Employee: Calendar (4:00pm - 12:00am)

Rule 3 - Card Permission 3: Shift 1-A Employee + Training: Calendar (8:00am - 4:00pm) +
Training (3:00pm - 5:00pm)

Rule 4 - Card Permission 4: Shift 2-B Employee + Training: Calendar (4:00pm - 12:00am) +
Training (3:00pm - 5:00pm)
In this case, if any of the rules change again, we will need to create yet another *Card Permission* rule.
This can become complicated and difficult to manage over time.

## 3.0 DEFINING OPERATOR ROLES

Using the role management feature, you can create a series of roles, each with customized rights and
privileges, that can be assigned to system users. These roles strictly control which functions a specific
user is allowed to perform within the system. In most cases, the user rights assigned to system
administrators who setup and maintain the database will be significantly different from the user rights
assigned to general operators, such as security guards.
For example, you might want to restrict system setup features, such as creating credentials and person
types, to users that have been given an administrator role. All other roles would not be given the
necessary rights to access these functions. When users who have been assigned non-administrator roles
login to the system, the icons related to the restricted functions will not appear on their screen,
effectively preventing them from viewing or modifying these aspects of the system. Once roles have
been defined, they can be assigned to registered individuals using the *Personnel* features of the
system.


Range

assigning person numbers and the total number of person numbers available. For example, if
you set the first value field to “100”, the system will assign new numbers for this person type
starting from 100 (100, 101, 102, etc.). If you wanted the system to stop assigning numbers
once it reached 1000, you would set “1000” in the second value field.

![Administration Panel image 401](./images/starwatch-sms-user-guide-administration-panel-rev-10_401.png)


Number of Characters
Using the Number of character positions field, you can set the number of characters used for
each person number. For example, if you set the starting Range as “100” and the Number of
character positions as 9, the system will assign numbers starting from 000000100 (000000100,
000000101, 000000102, etc.).

alarms occur.

![Administration Panel image 642](./images/starwatch-sms-user-guide-administration-panel-rev-10_642.png)

To launch this function, click on the
icon located on the *Procedures* segment of the
*Admin* list on the left side of the screen.

![Administration Panel image 643](./images/starwatch-sms-user-guide-administration-panel-rev-10_643.png)

![Administration Panel image 644](./images/starwatch-sms-user-guide-administration-panel-rev-10_644.png)

To add a group, click on the
button located at the top left of the work area. A new group will
be added to the *Email Group* list. Double-click on the new group to rename it.

![Administration Panel image 645](./images/starwatch-sms-user-guide-administration-panel-rev-10_645.png)

You can delete the group by selecting the group and then clicking on the
button.

![Administration Panel image 646](./images/starwatch-sms-user-guide-administration-panel-rev-10_646.png)

![Administration Panel image 647](./images/starwatch-sms-user-guide-administration-panel-rev-10_647.png)

To add a new email address to a selected group, click on the
button. The new email address
will be added to the *Email Address* list.

![Administration Panel image 648](./images/starwatch-sms-user-guide-administration-panel-rev-10_648.png)

You can delete an email address by selecting the address from the list and clicking the
button.

![Administration Panel image 649](./images/starwatch-sms-user-guide-administration-panel-rev-10_649.png)

To specify who can receive certain alarm notifications, select the checkboxes next to the email
addresses in that group.
